## AtomCAP 页面重构全流程（后端 → 前端 → 部署）

总体顺序：**Schema → Seed → Router → Page → SPA 剥离 → 提交部署**。

---

### 第 1 步：定义数据模型

**目录**：`prisma/schema.prisma`

- 在 `schema.prisma` 里声明 model（字段、索引、外键）。
- 有外键时要想清楚 `onDelete` 策略（`Cascade` / `Restrict` / `SetNull`）。例如 `Project.creatorId` 指向 `User`，`Task` / `Document` 随 `Project` 级联删除。
- 本地验证：`pnpm prisma db push`。

---

### 第 2 步：写Seed以在表中添加示例数据

**目录**：`prisma/seed.ts`

- 每张表一个 `seedXxx()` 函数，全部用 **"清空 + createMany" 或 upsert**，保证重复运行不产生脏数据。
- 涉及外键时，先 `upsert` 依赖记录（例如种子账号 `seed@atomcap.local`），只 `deleteMany` 属于种子用户的数据，避免误删真实业务数据。
- `main()` 按依赖顺序依次调用各 `seedXxx()`。
- 注册入口（只需配置一次，已就绪）：
  - `package.json` 里 `"prisma": { "seed": "tsx prisma/seed.ts" }`
  - `"scripts": { "db:seed": "prisma db seed" }`
  - `"build": "prisma db push --accept-data-loss && prisma db seed && ... next build"` — **每次部署自动执行**
- 本地跑一次：`pnpm db:seed`。

---

### 第 3 步：写 tRPC Router的后端逻辑

**目录**：`src/server/api/routers/<domain>.ts`

- 一个业务一个文件，例如 `dashboard.ts` / `project.ts` / `strategy.ts`。
- 每个 procedure 的三件套：
  1. `z.object({...})` 定义入参；查询无参用 `.query(async ({ ctx }) => ...)`。
  2. `ctx.db.xxx` 调 Prisma；权限相关用 `protectedProcedure`，公开用 `publicProcedure`。
  3. **返回 UI-ready 的 DTO**（在后端就 flatten 好，前端不做二次加工）。例如 `strategy.getAll` 直接返回 `{ owner: { id, name, initials }, totalInvest, returnRate }`。
- 注册路由：`src/server/api/root.ts` 里挂到 `appRouter` 下（如 `strategy: strategyRouter`）。
- 上下文与实例：`src/server/api/trpc.ts`（`createTRPCContext`、`initTRPC.create({ transformer: superjson })`）—— **transformer 必须和客户端一致**，否则部署后前端会统一报 "数据加载失败"。
- 客户端入口：`src/trpc/react.tsx`（`createTRPCReact<AppRouter>()` + `httpBatchLink({ transformer: superjson })`）。

---

### 第 4 步：写前端页面与其对应的路由

**目录**：`src/app/<route>/page.tsx`

结构固定按这个模板写（三页都一致）：

```tsx
"use client"
// 1) 登录守卫
const { status } = useSession()
useEffect(() => {
  if (status === "unauthenticated") router.replace("/login")
}, [status, router])

// 2) 拉数据（仅在已登录时 enable）
const { data, isLoading, isError } = api.<domain>.<proc>.useQuery(input, {
  enabled: status === "authenticated",
})

// 3) 写操作统一通过 mutation + invalidate 刷新
const utils = api.useUtils()
const mutation = api.<domain>.create.useMutation({
  onSuccess: () => utils.<domain>.<proc>.invalidate(),
})

// 4) 顶部导航：本页 return，其余 router.push 到兄弟路由或回 "/"
function handleTopNav(nav: TopNavKey) { ... }

// 5) loading / error / 数据三态 UI，主体只传 props 给纯展示组件
return (
  <div className="flex h-screen flex-col ...">
    <AppTopbar activeNav="<route>" onNavigate={handleTopNav} />
    <main>{isLoading ? ... : isError ? ... : <XxxView ... />}</main>
  </div>
)
```

**注意点**：

- Next 16 的 `useSearchParams` 需要 Suspense 包裹。如果只是在客户端读 URL 参数（如深链 `?projectId=`），直接在 `useEffect` 里读 `window.location.search` 更省事。
- 后端 DTO 与 UI 类型不一致时，在本文件顶部写一个 `adaptXxx()` 小函数 + 映射表。典型例子是 `src/app/strategies/page.tsx` 的 `ICON_MAP`：API 返 `iconName: string`，UI 要 `icon: ComponentType` + `iconBg`。

---

### 第 5 步：写纯展示组件（根据页面的需求，可选，不一定非要写）

**目录**：`src/components/pages/<domain>-*.tsx`（或 `src/components/` 下的 `*-view.tsx`）

- 组件**完全受控**：只接 props、不自己 fetch、不摸 `next-auth`、不摸 `next/navigation`。
- props 的类型从 tRPC 返回值 infer 出来或单独导出 `type DashboardOverviewDTO`、`type Project`、`type Strategy`，供 `page.tsx` 使用。
- 这样组件既能给 `/dashboard` 用，也能给将来的报表/预览页复用。

---

### 第 6 步：从老SPA（单页应用）架构的页面中剥离旧视图

**目录**：`src/app/page.tsx`

- 删掉对应的 `import`（如 `Dashboard` / `ProjectsGrid` / `StrategyCenter`）及其 `viewState` 分支。
- `handleTopNav` 里把相应 nav key 改成 `router.push("/<route>")`，而不是 `setView(...)`。
- 保留仍未独立拆分的视图（当前 `/` 只剩 `change-requests` / `project-detail` / `strategy-detail` 三个）。
- 深链参数：`/` 继续作为详情页宿主，其它路由跳详情时 `router.push("/?projectId=xxx")`。

---

### 第 7 步：提交与部署

- 推送分支：`git push origin <个人分支名>`。
- 部署时 `build` 会先 `prisma db push` → `prisma db seed` → `next build`，Seed 数据每次都重置到"干净演示态"。

## 一条典型重构的动作清单

以 "新增一个 `/reports` 报表中心" 为例：

1. `prisma/schema.prisma` 加 `Report` model → `pnpm prisma db push`。
2. `prisma/seed.ts` 加 `seedReports()`，`main()` 里调用。
3. `src/server/api/routers/report.ts` 写 `getAll` / `create`，在 `root.ts` 挂 `report: reportRouter`。
4. `src/components/pages/reports-grid.tsx` 写纯展示组件，导出 `type Report`。
5. `src/app/reports/page.tsx` 按模板写壳：`useSession` 守卫 + `api.report.getAll.useQuery` + `AppTopbar` + `<ReportsGrid>`。
6. 如果老 `src/app/page.tsx` 里有对应旧视图，按第 6 步剥离；`handleTopNav` 把 `reports` 改成 `router.push("/reports")`。
7. `git commit -m "新增报表中心页功能"` → `git push origin <个人分支名>`。
8. 看一下GitHub上Vercel的部署预览，确认 `/reports` 页面能正常访问且数据正确。
