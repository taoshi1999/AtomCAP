1. **修改请求每页的数量（可选，为了快速测试）**：
   打开 `app/page.tsx`，找到 `fetch('/api/projects?page=1&limit=10')` 和后续拉取数据的逻辑，把 `limit=10` 改成 `limit=2`。
2. **通过控制台 (F12) 观察 Network 网络请求**：

3. **滑动触发验证**：
   - 往下滚动页面直到滑到列表最底部。
   - 此时观察 Network 面板，系统会自动发出了一条新请求：`/api/projects?page=2&limit=10`。
   - 页面底部会短暂出现加载小圈圈，然后新卡片被拼接到界面上。这就证明了前后端配合生效了。

## 代码修改详情与逻辑解析

本次改造总共修改了 **5 个地方**，分为**后端**、**数据层**和**UI层**三大块。

### 1. 后端接口改造 (对应 `app/api/projects/route.ts` 和 `app/api/strategies/route.ts`)
* **修改逻辑**：从之前的单纯执行 `prisma.project.findMany()` 没有任何限制，变成了根据前端传过来的 `page` 和 `limit` 参数来计算偏移量进行部分提取。
* **核心代码**：
  ```typescript
  export async function GET(request: Request) { // 【修改点1】：原先的方法没有参数，现在接收 request 对象以便能够提取 URL 中的参数
    try {
      // 【修改点2】：解析每次请求附带的分页信息
      const { searchParams } = new URL(request.url)
      const page = parseInt(searchParams.get('page') || '1') // 当前请求的页码，默认 1
      const limit = parseInt(searchParams.get('limit') || '10') // 每次请求多少条数据，默认 10 条
      
      // 【修改点3】：通过 (当前页数 - 1) * 每页条数 来计算要在数据库里跳过 (skip) 多少条
      const skip = (page - 1) * limit
  
      // 【修改点4】：执行查询：利用 skip 和 take 纯数据库层面做截停，而不是把 100 遍数据查完再在前端分
      const data = await prisma.project.findMany({
        orderBy: { createdAt: 'desc' },
        skip: skip,   // 告诉数据库忽略前面已经下发给用户的数据
        take: limit,  // 只要拿当前所需量的数据片段
      })
      // ...后续返回数据逻辑不变
  ```
* **目的**：保护数据库和网络带宽，精确给出所请求窗口的数据片段。

### 2. 前端数据状态与调度改造 (对应 `app/page.tsx`)
* **修改逻辑**：为“项目”和“策略”两大模块分别引入分页的三个状态：当前页码(`page`)，是否还有更多数据(`hasMore`)，是否正在请求(`isLoadingMore`)。
* **核心代码**：
  ```typescript
  // 【新增状态】：跟踪当前是第几页、是不是请求到底了、防止网络延迟引起的重复请求
  const [projectsPage, setProjectsPage] = useState(1);
  const [hasMoreProjects, setHasMoreProjects] = useState(true);
  const [isLoadingMore, setIsLoadingMore] = useState(false);
  
  // 【修改点】：初始加载第一页数据，并在 useEffect 中增加判定逻辑
  useEffect(() => {
    // 首次进入时，只请求第一页的数据并设置 Limit 大小
    const [projectsResponse] = await Promise.all([
      fetch('/api/projects?page=1&limit=10')
    ]);
    const fetchedProjects = await projectsResponse.json();
    setProjects(fetchedProjects);
    // 若第一页请求回来的数据就不满 10 条，说明库里这点数据已经到底了
    if (fetchedProjects.length < 10) { setHasMoreProjects(false); }
  }, [])
  
  // 【新增方法】：提供给下拉框专用的获取下一页的方法
  const loadMoreProjects = async () => {
      // 如果正在加载（防止狂滚鼠标发几十次请求）或者没数据了就退出
      if (isLoadingMore || !hasMoreProjects) return;
      setIsLoadingMore(true);
      
      const nextPage = projectsPage + 1; // 准备获取下一页
      const res = await fetch(`/api/projects?page=${nextPage}&limit=10`);
      
      // 【将新数据通过解构语法合并拼接，不再是覆盖旧数据】
      setProjects(prev => [...prev, ...newProjects]);
      setProjectsPage(nextPage); // 更新当前页面索引
      setIsLoadingMore(false);
  }
  ```
* **目的**：集中管理数据，拼接每次滚动请求回来的下一页数据，并通过 props 把状态和方法传递给负责显示的 UI 组件。

### 3. UI 交互组件改造 (对应 `components/pages/projects-grid.tsx` 和 `strategies-grid.tsx`)
* **修改逻辑**：给组件增加了 `onLoadMore`、`hasMore` 和 `isLoadingMore` 等参数接收接口。利用 React 的 `useRef` 和 原生的 `IntersectionObserver` 监视器，监听列表里的**最后一张卡片**。
* **核心代码**：
  ```tsx
  // 【修改点1】：组件对外暴露触底事件，由外面 (app/page.tsx) 决定如何更新
  interface ProjectsGridProps {
    onLoadMore?: () => void
    hasMore?: boolean
    isLoadingMore?: boolean
  }

  // 【修改点2】：声明交叉观察器：用于监视某个元素是否被用户滚到屏幕视野内了
  const observer = useRef<IntersectionObserver | null>(null)
  const lastProjectElementRef = useCallback((node) => {
    if (isLoadingMore) return // 正在请求的间隙停止监控
    if (observer.current) observer.current.disconnect()
    
    observer.current = new IntersectionObserver((entries) => {
      // entries[0].isIntersecting 判断该卡片刚好露脸；如果是且后端还有数据，执行 onLoadMore 数据拼接操作
      if (entries[0].isIntersecting && hasMore && onLoadMore) { onLoadMore() }
    })
    if (node) observer.current.observe(node) // 将节点添加到观察者队列
  })
  
  // 【修改点3】：渲染列表的时，将这个 ref '摄像头' 始终挂载在当前数据的最后一张卡片上
  {filteredProjects.map((project, index) => (
      <button 
        ref={index === filteredProjects.length - 1 ? lastProjectElementRef : null}
      > {/* ...卡片内容 */} </button>
  ))}
  
  // 【修改点4】：正在读取时，在页面底部给出动画提示
  {isLoadingMore && <Loader2 className="h-6 w-6 animate-spin text-[#2563EB]" />}
  ```
* **目的**：免去用户手动点击“下一页”的麻烦，提供流畅的“无限滚动瀑布流”体验；同时底部根据 `isLoadingMore` 的状态按需显示菊花转圈加载的交互反馈（`Loader2` 组件）。
